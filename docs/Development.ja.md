# AI駆動開発ワークフロー

## 目的

このプロジェクトでは、複数のAIを使った開発フローによって、アイデアをレビュー済みで検証可能な変更へ落とし込みます。

目指しているのは完全自動化ではありません。目標は、次の流れを安定して回せることです。

- GitHub Issueとして作業を計画する
- Builder AIが実装する
- 独立した Reviewer AIがレビューする
- テスト結果とレビュー結果を次の計画に反映する

GitHub Issues は、このシステムの永続的な記憶として扱います。

## 役割

各AIは異なる責務を持ちます。

| Role | Primary Tool | Responsibility |
| ---- | ------------ | -------------- |
| Planner | Antigravity | 要求を分析し、Issueに分解し、次のアクションを提案する |
| Builder | Claude Code | 承認済みの作業を実装し、必要に応じてテストを追加・更新する |
| Reviewer | Cursor | 差分を独立してレビューし、リスク、回帰、テスト不足を指摘する |
| Validator | ローカルテスト + 手動確認 | ビルド、動作、ログが許容範囲かを確認する |

この分離によって、単一モデルが計画・実装・承認まで自己完結してしまうリスクを下げます。

## 開発ループ

```text
アイデアまたはIssue
  ->
計画 (Issueの起票)
  ->
実装 (機能ブランチ作成 & PR提出)
  ->
テスト (PR内での検証)
  ->
独立レビュー (PRに対するレビュー)
  ->
改善分析 (PRへのフィードバック修正 or マージ)
  ->
次Issueまたはクローズ
```

このワークフローは、実務的なPDCAサイクルに対応します。

| Phase | Owner | Main Output |
| ----- | ----- | ----------- |
| Plan | Antigravity | スコープと受け入れ条件が定義されたIssue |
| Do | Claude Code | コード変更と関連テスト |
| Check | テスト + Cursor | 検証結果とレビュー指摘 |
| Act | Antigravity | フォローアップIssue、優先度見直し、またはクローズ判断 |

## ハンドオフルール

各フェーズには、明確な入力と出力が必要です。原則として、実装からレビューへの受け渡しは **Pull Request (PR)** を通じて行われます。

| Phase | Input | Output | Done When |
| ----- | ----- | ------ | --------- |
| Planning | アイデア、バグ報告、レビュー指摘、または既存Issue | スコープ、制約、成功条件を含むGitHub Issue | 推測なしで実装に着手できる粒度になっている |
| Implementation | 承認済みIssue | コード差分と、必要ならテストやドキュメント | 変更がIssueと一致し、ビルドが通る |
| Testing | 実装差分 | テスト結果、ログ観察結果、動作メモ | 回帰確認が行われ、証拠が記録されている |
| Review | 差分とテスト証跡 | 指摘、リスク、テスト不足、改善提案 | go/no-go の判断が可能 |
| Improvement Analysis | レビュー結果とテスト結果 | フォローアップIssue、またはクローズ判断 | 次のアクションが明示されている |

## 実行ルール

### 1. 1 Issue 1 Outcome

各実装サイクルは、1つのIssue、または密接に関連する変更群に集中させます。

次の混在は避けます。

- 機能追加と無関係なリファクタ
- 挙動変更と推測ベースのクリーンアップ
- 独立した複数改善の同時投入

### 2. 最終ゲートは人間が持つ

AIはIssue、修正案、次の作業を提案してよいですが、最終判断は人間が行います。

- 新規作業の起票や優先順位付け
- 重要変更のマージ
- リリース可否
- ユーザー影響のある挙動変更の受け入れ

### 3. 低信頼の指摘で自動ループしない

テスト結果、ログ、レビュー指摘が曖昧な場合は、推測で次Issueを増やさず、一度止めて確認を取ります。

### 4. 新規Issueより既存Issueの更新を優先する

レビュー指摘が現在作業中の内容の延長線上にある場合は、別Issue化の利点が明確でない限り、まず既存Issueへ証拠を追記します。

### 5. レビュー指摘はPR内で修正を完結させる

Reviewフェーズで問題が指摘された（差し戻しが発生した）場合、原則として新しいIssueは立てません。現在のPR内で Builder（Claude Code）が修正コミットを追加し、再度 Testing と Review のループを回してマージ可能な状態を目指します。

## Issue に必要な内容

実装可能なIssueは、最低限次の問いに答えている必要があります。

- 何の問題を解決するのか
- なぜ今それが重要なのか
- スコープはどこまでか
- 何をスコープ外とするか
- どうやって成功を確認するか
- 再現手順や証拠はあるか

受け入れ条件は、具体的かつ検証可能であるべきです。

## テストと検証

テストは補助ではなく、ループの一部です。

### このプロジェクトでの最低限の検証

プロジェクト標準コマンドを使います。

```bash
./build.sh
./build.sh --run
./build.sh --install
```

変更を確認するのに必要な最小コマンドを選びます。ビルド確認だけなら `./build.sh` を優先し、挙動や統合確認が必要な場合のみ `--run` または `--install` を使います。

確認対象のログ:

```text
~/Library/Logs/KeyLens/app.log
```

### テストで見るべきこと

- 回帰検出
- アルゴリズムの正しさ
- エッジケース
- 失敗条件
- 必要に応じた性能面の影響

### 検証からIssueを作成・更新する条件

次のいずれかが見つかった場合は、Issueを新規作成または更新します。

- 再現可能な失敗
- 計測可能な性能劣化
- リスクの高い経路に対するテスト不足
- リリース判断を妨げる曖昧な挙動

## レビューの期待値

独立レビューでは、次を優先的に確認します。

- 挙動回帰
- アーキテクチャまたは保守性のリスク
- テスト不足
- 隠れた前提
- 性能上の懸念

レビューの成果物は、抽象的な点数ではなく、次に動ける指摘であるべきです。

## ガードレール

このワークフローを有効かつ制御可能に保つため、次を守ります。

- 重複Issueを自動生成しない
- 証拠なしにフォローアップIssueを作らない
- 推測ベースのAI提案を確定バグとして扱わない
- 次アクションが明示されるまでループを閉じない

複数のAIの見解が食い違った場合は、その不一致自体を Issue の履歴として残します。

## 今後の発展

将来的な拡張候補は次の通りです。

- エッジケースや失敗系に対するAI生成テスト
- メトリクスベースの性能チェック
- Issue のクラスタリングと重複検知の改善
- 検証済みの指摘からのドラフトIssue自動生成

ただし、これらは現在のループが安定してから段階的に導入します。

長期的なゴールは、AIが計画・実装・レビュー・改善提案を継続的に支援しつつ、GitHub Issues がプロジェクトの記憶を保持し、最終判断は人間が持つ自己改善型の開発環境を作ることです。
